# Keaton — History

## Core Context

**Project:** Cookbook Web Application  
**User:** Rick Wilkerson  
**Stack:** ASP.NET Core Razor Pages, EF Core, ASP.NET Core Identity, HTMX, Bootstrap 5, PostgreSQL, .NET Aspire (orchestration), Playwright (UI testing)

**Domain Model Summary:**
- `ApplicationUser` (extends IdentityUser) — owns cookbooks and recipes
- `Cookbook` — Id (int), OwnerId, Name, Description, PublicId (unique), Slug, CreatedAt, UpdatedAt
- `Recipe` — Id (int), OwnerId, Title, Description, Ingredients, Instructions, PrepTime, CookTime, Servings, PublicId (unique), Slug, OriginalRecipeId (for clones), CreatedAt, UpdatedAt
- `CookbookRecipe` — join table (CookbookId, RecipeId, SortOrder), composite PK
- `Share` — grants Read or Update permission on a Cookbook or Recipe to another user

**Key Patterns:**
- URLs use `{publicId}/{slug}` — never internal int IDs
- publicId: random Base62, 8–12 chars, stable, unique per entity
- Slug: derived from name/title, lowercase with hyphens, NOT required to be unique
- Authorization: ownership OR Share record grants access
- Cookbook share implies read access to contained recipes

**URL Routes:**
- `/cookbooks` — list my cookbooks
- `/cookbooks/{publicId}/{slug}` — cookbook details
- `/recipes/{publicId}/{slug}` — recipe details

**Architecture: Vertical Slice + MediatR**
- Application logic lives in `Features/` folder, NOT in PageModels
- Each feature slice: `Features/{Domain}/{FeatureName}/{Command|Query}.cs`, `Handler.cs`, `Response.cs`
- Razor Pages inject `IMediator` and call `Send()` only — no business logic in .cshtml.cs
- Handlers are the unit-testable seam; testable without the web layer
- Future clients (API, CLI, mobile) dispatch the same MediatR requests
- Key slices to design: CreateCookbook, GetCookbook, ListCookbooks, AddRecipeToCookbook, ShareCookbook, CreateRecipe, GetRecipe, CloneRecipe, ShareRecipe

## Learnings

### 2026-03-05: Admin CLI Plan Review Complete

**Decision:** Plan is architecturally sound; recommended for implementation pending 6 clarifying questions from Rick (auth strategy, scope, admin identity, shared layer, Hex1b spike, authorization model).

**Key architectural insights:**
- CLI must be thin presentation layer only; admin logic goes in dedicated `Recipe.AdminApi` backend. This mirrors how Razor Pages work (thin + MediatR handlers).
- Admin API should use vertical slices + MediatR (same pattern as Recipe.Web) for consistency and future client reuse (web, mobile, CLI all dispatch same handlers).
- **Two paths for reuse:** Path A (recommended for current scale): `Recipe.AdminApi` references `Recipe.Web.Features` directly. Path B: Extract `Recipe.ApplicationCore` early if expecting multiple clients. Start with Path A to avoid over-architecture.
- **Auth strategy is critical decision:** OIDC device flow recommended (no browser dependency, token caching in CLI) but requires confirmation before API build. Fallback for local dev: scoped env-var token.
- **First-phase scope must be explicit:** Constrain v1 to one workflow (e.g., search users, enable/disable, assign role) to ship fast. Avoid bulk ops, audit logs, password reset in v1.
- **Aspire integration is straightforward:** Add `Recipe.AdminApi` as a resource in `AppHost.cs` early; de-risks before heavy implementation.
- **Test strategy for TUI is lighter:** Focus on handler + endpoint tests; minimal TUI integration tests (Playwright doesn't work for TUI).
